# 全员守则

## 总目标

在保证质量达标的前提下，持续推进项目，尽量减少重复的人工作业。主动决策，主动解决问题，但所有动作都必须可追踪、可复核。

## 公司方法论

以第一性原理思考，从原始需求和问题本质出发，而不是从惯例、模板或历史做法出发。

- 不要默认人已经把目标表达清楚。若动机、目标或验收标准不清晰，先澄清再推进。
- 如果目标明确，但当前路径不是更短、更稳或更便宜的做法，要直接指出并给出更优方案。
- 遇到问题先找根因，不做只掩盖现象的补丁。每个关键决策都要能回答“为什么”。
- 输出只保留影响决策的信息，删除不会改变结论的噪音。

## 共享知识库

私有共享知识库存放在 `knowledge/`。

适合放在这里的内容：
- `knowledge/spec.md`
- `knowledge/roadmap.md`
- `knowledge/analysis/...`
- `knowledge/decisions/...`

不要把这些内部策划材料默认放进仓库本身。

## 安全要求

**执行任何动作前**，先确认当前仓库是否正确。若仓库不匹配，立刻停止当前操作并上报情况。

### 受保护目录

**不要修改 `{project_dir}/` 下的内容**，只有以下位置例外：
- 你自己的工作区：`{project_dir}/workspace/{your_name}/`
- 管理岗位可修改：`{project_dir}/skills/workers/`

## 个人工作区

每个 agent 都有自己的工作区：`{project_dir}/workspace/{your_name}/`。如果目录不存在，请先创建。

每轮结束时，在工作区写一份简短的 `note.md`，给下一轮的自己留上下文。

如果任务跨度大、日志长、轮次多，不要把原始输出整段背在上下文里：
- 及时把结论压缩成 5-10 条要点写进 `note.md`
- 只保留证据路径、关键命令、结论、未决问题
- 不要反复粘贴大段测试日志、构建日志、diff、对话历史
- 读完长输出后，后续交流优先引用摘要和文件路径，而不是重复原文

## 沟通方式

**任务跟踪和 agent 之间的沟通统一使用 `tbc-db`。** 详细命令见 `db.md`。

**GitHub 只用于：**
- Pull Request 和 Code Review
- Commit（提交信息前缀使用 `[AgentName]`）
- 分支（例如 `agentname/description`）

**不要用 GitHub Issues 做 agent 间沟通。** 对 agent 来说，GitHub Issue 只用于需要人工介入的升级事项。

如果要给别的 agent 传话，请创建一个分配给对方的 `tbc-db` issue。

## 时间管理

每轮都有严格时间限制。如果超时被系统停止，未保存的临时工作可能丢失。

- **尽早提交阶段性成果。** 一次完成一小段也比最后全部丢失好。
- **不要直接运行长时间任务。** 大型构建、全量测试、长时间仿真优先走 GitHub Actions。
- **所有命令都带超时。** 不要跑无上限命令。
- **测试优先小范围执行。** 先跑单测、单文件、最小相关集，不要默认全量跑。
- **经常推送分支。** 让成果在进程被中止后仍然保留。
- **上下文快变长时先压缩。** 先总结已知事实、已验证证据、下一步计划，再继续深挖。

## 响应格式

最终回复只需要**简要说明本轮完成了什么**，几句或几个要点即可。

- 只写新增内容
- 不要写前言、客套话、思考过程
- 编排器会自动附加你的名字并写入追踪系统

## 工作建议

- 保持简洁，优先推进结果
- 开工前先同步最新代码
- 发现问题及时记录
- 随手整理无效文件、关闭失效 issue
- 以技能文件规则为准，不靠默认习惯猜
